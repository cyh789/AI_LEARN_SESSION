# Feature Specification: 증권 시장 스냅샷 웹앱

## User Scenarios & Testing (mandatory)

### User Story 1: 초기 진입 시 현재 시장 상태를 본다

- As: 장 시작 직후 또는 장중에 시장 흐름을 빨리 확인하고 싶은 사용자로서
- I: 앱에 들어오자마자 코스피, 코스닥, 주요 종목의 최신 스냅샷을 보고 싶습니다
- So: 현재 시장 분위기를 즉시 파악하기 위해

#### Acceptance Scenarios

Scenario 1: **초기 스냅샷 로딩 성공**

- Given: 서버 캐시에 `KOSPI`, `KOSDAQ`, `KOSPI200`, `005930`, `000660` 최신 데이터가 있을 때
- When: 사용자가 앱에 처음 진입하면
- Then: 브라우저는 `/api/market/snapshot` 응답을 받아 카드 목록을 렌더링합니다
- Then: 각 카드에는 자산명, 현재가, 등락값, 등락률, 마지막 업데이트 시각이 표시됩니다
- Then: 상승/하락/보합 상태가 색상 또는 배지로 구분됩니다

### User Story 2: 새로고침 없이 시장 변화를 계속 본다

- As: 화면을 계속 켜 두고 시장을 보는 사용자로서
- I: 페이지 전체를 새로고침하지 않아도 최신 가격 변화를 보고 싶습니다
- So: 장중 흐름을 끊김 없이 추적하기 위해

#### Acceptance Scenarios

Scenario 1: **스트림 업데이트 반영**

- Given: 브라우저가 `/api/market/stream`에 연결되어 있을 때
- When: 서버가 외부 공급자로부터 `KOSPI` 또는 `KOSDAQ`의 새 체결 데이터를 받으면
- Then: 해당 카드의 가격, 등락값, 등락률, 업데이트 시각이 전체 페이지 새로고침 없이 갱신됩니다

Scenario 2: **연결 끊김 후 복구**

- Given: 브라우저가 스트림 연결을 사용 중일 때
- When: 네트워크 문제 또는 서버 재시작으로 스트림 연결이 일시적으로 끊기면
- Then: 화면에 `재연결 중` 상태가 표시됩니다
- Then: 연결이 복구되면 최신 데이터로 자동 동기화됩니다

### User Story 3: 관심 자산과 개인 관점을 저장한다

- As: 반복적으로 같은 자산을 보는 사용자로서
- I: 관심 자산과 `Long`, `Short`, `Watch` 상태, 메모를 저장하고 싶습니다
- So: 내 관찰 대상과 판단을 다음 접속 때도 이어서 보기 위해

#### Acceptance Scenarios

Scenario 1: **관심 자산 저장**

- Given: 사용자가 `KOSPI` 카드를 보고 있을 때
- When: pin 버튼을 누르면
- Then: `관심 자산` 섹션에 `KOSPI`가 추가됩니다
- Then: 새로고침 후에도 `KOSPI`가 유지됩니다

Scenario 2: **개인 관점 저장**

- Given: 사용자가 `005930` 카드를 보고 있을 때
- When: `Watch`를 선택하고 메모 `외국인 수급 확인 필요`를 저장하면
- Then: 해당 카드에 `Watch` 상태와 메모가 표시됩니다
- Then: 새로고침 후에도 저장값이 복원됩니다

### User Story 4: 외부 시세 공급자 장애가 나도 앱 상태를 이해할 수 있다

- As: 데이터가 비거나 멈춘 이유를 알고 싶은 사용자로서
- I: 공급자 장애나 지연 상황을 화면에서 확인하고 싶습니다
- So: 잘못된 가격을 최신 가격으로 오해하지 않기 위해

#### Acceptance Scenarios

Scenario 1: **외부 공급자 지연**

- Given: 서버가 외부 공급자에서 새 데이터를 일정 시간 이상 받지 못했을 때
- When: 브라우저가 앱을 보고 있으면
- Then: 화면에 `지연됨` 또는 `stale` 상태가 표시됩니다
- Then: 마지막 수신 시각이 함께 표시됩니다

Scenario 2: **초기 스냅샷도 불러오지 못함**

- Given: 서버가 외부 공급자 인증 실패 또는 연결 실패 상태일 때
- When: 사용자가 앱에 처음 접속하면
- Then: 빈 카드 목록 대신 오류 배너와 재시도 안내가 표시됩니다

## Functional Requirements (mandatory)

- FR-1: MUST 시장 데이터 취득은 서버가 외부 API를 호출하는 방식으로 수행해야 한다
- FR-2: MUST 브라우저는 외부 증권 API가 아니라 우리 서버의 `/api/market/snapshot`과 `/api/market/stream`만 호출해야 한다
- FR-3: MUST 서버는 외부 공급자 인증 정보를 서버 환경변수로 관리해야 한다
- FR-4: MUST 서버는 최소 `KOSPI`, `KOSDAQ`, `KOSPI200`의 최신 스냅샷을 제공해야 한다
- FR-5: MUST 서버는 대표 종목 몇 개 이상을 함께 제공해야 한다
- FR-6: MUST 서버는 외부 공급자 원본 데이터를 내부 공통 포맷으로 정규화해야 한다
- FR-7: MUST 서버는 최신 시장 데이터를 메모리 캐시에 유지해야 한다
- FR-8: MUST 브라우저 첫 진입 시 스냅샷 REST 응답으로 초기 화면을 렌더링해야 한다
- FR-9: MUST 브라우저는 스트림 연결을 통해 변경분을 새로고침 없이 반영해야 한다
- FR-10: MUST 화면에 연결 상태(`연결 중`, `정상`, `재연결 중`, `지연`, `오류`)를 표시해야 한다
- FR-11: MUST 자산 카드에 자산명, 현재가, 등락값, 등락률, 마지막 업데이트 시각을 표시해야 한다
- FR-12: MUST 상승/하락/보합 상태를 명확히 구분하는 시각 표현을 제공해야 한다
- FR-13: MUST 사용자가 자산을 관심 목록에 추가/제거할 수 있어야 한다
- FR-14: MUST 관심 목록 상태를 브라우저 `localStorage`에 저장해야 한다
- FR-15: MUST 사용자가 자산별로 `Long`, `Short`, `Watch` 상태와 짧은 메모를 저장할 수 있어야 한다
- FR-16: MUST 저장한 관심 자산, 상태, 메모를 새로고침 후 복원해야 한다
- FR-17: MUST 서버는 외부 스트림이 끊기면 자동 재연결을 시도해야 한다
- FR-18: MUST 외부 공급자 장애 또는 데이터 지연 시 stale 상태를 계산해 내려줘야 한다
- FR-19: MUST 실시간 상태 판단 기준을 숫자로 고정해야 한다
- FR-20: MUST `fresh`는 마지막 정상 시세 수신 후 `3초 미만`, `stale`는 `15초 이상`, `disconnected`는 `60초 이상`으로 정의해야 한다
- FR-21: MUST 화면과 API 모두에서 시장 상태를 `preopen`, `open`, `afterhours`, `closed`, `weekend` 중 하나로 표시해야 한다
- FR-22: MUST 국내 정규장, 장마감, 야간선물, 주말 상태를 구분해 표시해야 한다
- FR-23: MUST 공급자 장애 시 마지막 정상 데이터와 마지막 수신 시각을 유지한 채 `stale` 또는 `disconnected` 상태를 내려줘야 한다
- FR-24: MUST 공급자 장애 시 가격 값을 `0`이나 빈값으로 덮어쓰지 않아야 한다
- FR-25: MUST 공급자 복구 후 첫 정상 수신 시 stale 상태를 해제하고 최신 값으로 동기화해야 한다
- FR-26: SHOULD 수동 새로고침 버튼을 제공해야 한다
- FR-27: SHOULD 장 마감 또는 비거래 시간에도 마지막 수신 데이터와 시장 상태를 표시해야 한다
- FR-28: SHOULD 2차 단계에서 글로벌 지수와 크립토 공급자를 추가할 수 있도록 공급자 어댑터 구조를 분리해야 한다

## Constraints (mandatory)

- CON-1: MUST 프론트엔드는 `HTML`, `CSS`, `JavaScript` 중심의 파일 기반 구조를 유지해야 한다
- CON-2: MUST 데이터 수집을 위해 별도의 Node 서버 계층을 둬야 한다
- CON-3: MUST 외부 API 키, 시크릿, 토큰을 브라우저에 노출하면 안 된다
- CON-4: MUST 외부 공급자의 호출 제한, 연결 제한, 사용 약관을 준수해야 한다
- CON-5: MUST `KRX` 공개 지연 데이터를 `거의 실시간` 핵심 소스로 사용하지 않아야 한다
- CON-6: MUST 서버 장애 시 프론트엔드가 무한 로딩에 빠지지 않고 오류 상태를 표시해야 한다
- CON-7: MUST 사용자별 개인 상태 저장은 1차 버전에서 브라우저 `localStorage`만 사용해야 한다
- CON-8: SHOULD 브라우저 스트림은 1차 버전에서 단방향 전송에 적합한 `SSE`를 우선 사용해야 한다
- CON-9: SHOULD 공급자별 데이터 포맷 차이를 `provider adapter` 계층에서 흡수해야 한다

## Real-Time Operating Rules (mandatory)

- RTR-1: MUST `fresh` 상태는 마지막 정상 시세 수신 시각이 현재 시각 기준 `3초 미만` 차이일 때만 부여한다
- RTR-2: MUST `stale` 상태는 마지막 정상 시세 수신 시각이 현재 시각 기준 `15초 이상` 차이일 때 부여한다
- RTR-3: MUST `disconnected` 상태는 마지막 정상 시세 수신 시각이 현재 시각 기준 `60초 이상` 차이이거나 스트림 연결이 완전히 끊긴 경우 부여한다
- RTR-4: MUST `3초 이상 15초 미만` 구간은 `delayed` 또는 `connecting`으로 처리할 수 있으나, `fresh`로 표시하면 안 된다
- RTR-5: MUST `open`은 국내 정규 거래시간 동안 사용한다
- RTR-6: MUST `closed`는 장마감 상태를 의미하며, 국내 정규장이 종료되고 별도 야간 데이터 소스가 활성화되지 않은 시간에 사용한다
- RTR-7: MUST `afterhours`는 코스피200 야간선물 등 야간 데이터 소스가 활성화된 시간에 사용한다
- RTR-8: MUST `weekend`는 토요일과 일요일, 또는 공급자 기준 비거래 휴장일에 사용한다
- RTR-9: MUST 공급자 장애 시 마지막 정상 가격과 마지막 정상 수신 시각을 유지하고 상태만 `stale` 또는 `disconnected`로 변경한다
- RTR-10: MUST 공급자 장애 중에는 추정값이나 임의 보정값을 새 시세처럼 표시하지 않는다
- RTR-11: MUST 공급자 복구 후 첫 정상 수신 데이터가 들어오면 `updatedAt`, 가격, 등락값, 등락률을 즉시 갱신한다
- RTR-12: MUST 프론트엔드는 상태 배지와 함께 `마지막 수신 시각`을 항상 표시해야 한다

## Success Criteria (mandatory)

- SC-1: 앱 첫 화면에서 `/api/market/snapshot` 응답으로 코스피, 코스닥, 코스피200, 대표 종목 카드가 정상 렌더링된다
- SC-2: 외부 공급자에서 새 가격이 들어오면 전체 새로고침 없이 화면 수치가 갱신된다
- SC-3: 스트림 연결이 끊기면 `재연결 중` 또는 `지연` 상태가 사용자에게 보인다
- SC-4: 관심 자산 추가/제거와 `Long`, `Short`, `Watch`, 메모 저장이 새로고침 후에도 유지된다
- SC-5: 외부 공급자 인증 실패 또는 연결 실패 시 빈 화면이 아니라 오류 상태가 표시된다
- SC-6: 모바일 폭 약 `390px`과 데스크톱 폭 약 `1280px`에서 핵심 가격 정보와 연결 상태가 깨지지 않는다
- SC-7: 마지막 수신 후 `3초 미만`이면 `fresh`, `15초 이상`이면 `stale`, `60초 이상`이면 `disconnected` 상태가 일관되게 표시된다
- SC-8: 장중, 장마감, 야간선물, 주말 상태가 각 시간대에 맞게 구분되어 표시된다
- SC-9: 공급자 장애 시 마지막 정상 가격은 유지되고 상태와 마지막 수신 시각만 변경된다
