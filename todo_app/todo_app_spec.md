# Todo 앱 정의서

## 1) 목표
- 사용자가 할 일을 빠르게 기록하고 관리할 수 있는 간단한 Todo 앱을 만든다.
- 개인 생산성 도구로 시작하되, 이후 협업 기능까지 확장 가능한 구조를 고려한다.
- 웹 브라우저와 모바일 환경에서 모두 무리 없이 사용할 수 있도록 설계한다.

## 2) 핵심 사용자 시나리오
- 사용자는 할 일을 즉시 추가할 수 있어야 한다.
- 사용자는 할 일의 완료 여부를 체크할 수 있어야 한다.
- 사용자는 진행 중인 작업과 완료된 작업을 구분해서 볼 수 있어야 한다.
- 사용자는 중요한 할 일을 우선순위나 마감일 기준으로 관리할 수 있어야 한다.

## 3) 기본 기능
### 할 일 관리
- Todo 추가
- Todo 수정
- Todo 삭제
- Todo 완료/미완료 상태 변경

### 목록 조회
- 전체 Todo 목록 조회
- 완료된 Todo 조회
- 미완료 Todo 조회
- 우선순위별 필터링
- 마감일 기준 정렬

### 부가 정보
- 제목
- 상세 설명
- 생성일
- 마감일
- 우선순위(높음/중간/낮음)
- 태그

### 사용성
- 모바일 반응형 UI
- 입력 폼 검증
- 로딩/에러 상태 표시
- 빈 목록 상태 안내

## 4) 있으면 좋은 확장 기능
- 카테고리별 Todo 관리
- 검색 기능
- 반복 일정 기능
- 캘린더 보기
- 알림 기능
- 소셜 로그인
- 다크 모드
- 협업용 공유 리스트

## 5) 기술 스택 제안
### 프론트엔드
- `HTML`
  - 기본 화면 구조를 단순하고 직관적으로 구성
- `CSS`
  - 화면 스타일링과 반응형 레이아웃 구성
- `JavaScript`
  - Todo 추가, 수정, 삭제, 필터링 등 동적 기능 구현
- `Bootstrap` 또는 기본 CSS
  - 빠르게 UI를 구성하거나 가볍게 직접 스타일링 가능

### 백엔드
- `Node.js + Express`
  - HTML/JavaScript 기반 프론트엔드와 연결하기 쉽고 구조가 단순함
  - 초기 CRUD API를 빠르게 만들기 좋음

### 데이터베이스
- `PostgreSQL`
  - 정형 데이터 관리에 적합하고 확장성이 좋음
- `Prisma`
  - 스키마 관리와 ORM 사용 편의성 확보

### 인증
- 초기 버전은 인증 없이 시작
- 이후 필요 시 `Express Session` 또는 OAuth 기반 로그인 추가

### 배포 및 운영
- `Vercel`, `Netlify`, 또는 `GitHub Pages`
  - 정적 프론트엔드 배포가 간단함
- `Railway` 또는 `Render`
  - Express 서버와 데이터베이스 운영에 적합함

### 테스트
- `Jest`
  - JavaScript 로직 단위 테스트
- `Playwright`
  - 브라우저 기반 E2E 테스트

## 6) 권장 아키텍처
- 초기 버전은 `HTML + CSS + JavaScript`로 프론트엔드를 구성한다.
- 백엔드는 `Node.js + Express`로 간단한 REST API를 제공한다.
- 데이터는 PostgreSQL에 저장하고 Prisma 또는 기본 SQL 쿼리 방식으로 접근한다.
- 인증이 필요 없는 MVP부터 시작한 뒤, 사용자 계정 기능을 2차 단계에서 추가한다.
- 프론트엔드, API, DB 계층을 분리해 이후 기능 확장 시 영향 범위를 줄인다.

## 7) MVP 범위
- Todo 추가/수정/삭제
- 완료 상태 토글
- 전체/완료/미완료 필터
- 우선순위와 마감일 입력
- 반응형 UI
- 로컬 또는 DB 저장

## 8) 화면 구성 예시
- `메인 화면`
  - Todo 입력 영역
  - 필터 탭
  - Todo 리스트
- `상세/수정 화면`
  - 제목, 설명, 우선순위, 마감일 수정
- `설정 화면(선택)`
  - 테마 설정
  - 계정 설정

## 9) 개발 단계 제안
1. UI 와이어프레임 구성
2. Todo CRUD 기능 구현
3. 필터/정렬 기능 구현
4. DB 연동
5. 인증 추가
6. 테스트 및 배포

## 10) 최종 제안
- 첫 버전은 `HTML + CSS + JavaScript + Node.js + Express + PostgreSQL` 조합을 권장한다.
- 이유는 구조가 단순하고 학습 비용이 낮으며, Todo 앱 같은 CRUD 서비스에 적합하기 때문이다.
- 이후 사용자 수가 늘어나면 인증, 협업, 알림 기능을 순차적으로 확장한다.
